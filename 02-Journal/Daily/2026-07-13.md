# 💡 Key Learnings

## Why Searching Belongs in the Service Layer

Today's implementation reinforced the importance of separating business logic from user interaction.

The CLI should only collect user input and display results.

The Service Layer is responsible for deciding **how** projects are searched.

Keeping the search logic inside the Service Layer makes it reusable for future interfaces such as a web application, REST API, or desktop application without rewriting the search functionality.

---

## Code Reusability

One of the biggest lessons today was understanding the value of reusing existing code.

Instead of writing another function to read the JSON file, I reused the existing `load_projects()` function.

This reduced duplication, simplified maintenance, and ensured that any future changes to file handling only need to be made in one place.

---

## Linear Search

Today's feature introduced the concept of **Linear Search**.

I learned that the application checks each project one by one until it finds a match.

Although this approach has a time complexity of **O(n)**, it is simple and appropriate for a small collection of projects.

As the application grows and stores larger amounts of data, more efficient search techniques or databases can be introduced.
