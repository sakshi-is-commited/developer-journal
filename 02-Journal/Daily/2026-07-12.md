# 💡 Key Learnings

## Separation of Storage, Service and CLI

Today's feature clearly demonstrated the value of separating different responsibilities within an application.

The **Storage Layer** focused only on reading project data from the JSON file.

The **Service Layer** handled the business logic by requesting project data from storage and preparing it for the application.

The **CLI Layer** was responsible only for interacting with the user and displaying information.

Because each layer had a single responsibility, implementing the View Projects feature required only small changes within the appropriate module instead of modifying the entire application.

This reinforced the importance of modular software design.

---

## Displaying Data vs Saving Data

Saving data and displaying data involve different responsibilities.

When saving data, the application validates user input and writes structured information into persistent storage.

Displaying data is the opposite process. Instead of collecting information, the application retrieves stored data and presents it in a clean, readable format for the user.

Understanding this distinction helped me appreciate how applications manage the flow of data between users and storage.

---

## Exception Handling

Today's implementation introduced the importance of exception handling.

I learned that applications should never assume external resources, such as files, are always available or correctly formatted.

Using exception handling allows the application to recover gracefully from unexpected situations instead of terminating abruptly.

This improves both the reliability of the software and the overall user experience.

# 🤔 Challenges Faced

One of the challenges was designing the output so that multiple projects remained easy to read within the terminal.

Another challenge was deciding where different pieces of logic belonged.

It was tempting to perform file operations directly inside the CLI, but keeping storage operations inside the Storage Layer resulted in a much cleaner architecture.

Implementing proper handling for empty project lists also required careful consideration to provide meaningful feedback to the user.
