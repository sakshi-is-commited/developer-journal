# Daily Journal — 23 July 2026

## Sprint

Sprint 03 — PostgreSQL Migration

---

## Today's Goal

Replace the JSON-based "Add Project" functionality with PostgreSQL while keeping the application's architecture clean and modular.

---

## What I Worked On Today

Today's focus was migrating the first CRUD operation from JSON storage to PostgreSQL.

I implemented the Repository Pattern by creating the `add_project()` method inside `project_repository.py`. The repository is responsible for establishing the database connection, executing parameterized SQL INSERT statements, committing transactions, and safely closing database resources.

Next, I updated the Service Layer so that it delegates project creation to the repository instead of interacting with JSON storage directly. One of the biggest advantages of the project's layered architecture became clear today—the CLI required almost no changes because the storage implementation was abstracted behind the service layer.

Finally, I tested the migration by creating new projects through the CLI and verified the inserted records directly using PostgreSQL.

---

## Challenges Faced

During implementation I encountered two important issues.

The first was:

```
ModuleNotFoundError: No module named 'src'
```

This occurred because I executed the repository file directly instead of running it as a Python module. After understanding Python's package structure, I corrected the execution method.

The second issue was:

```
Database Error:
not all arguments converted during string formatting
```

This was caused by incorrectly formatting SQL parameters. After reviewing the parameterized query syntax, I corrected the SQL execution and verified that projects were inserted successfully into PostgreSQL.

Both issues helped me understand Python module execution and SQL parameter binding more deeply.

## Key Learnings

Today I learned:

- How to insert records into PostgreSQL using `psycopg2`.
- Why parameterized SQL queries are important for security and correctness.
- How database transactions work using `commit()` and `rollback()`.
- How the Repository Pattern keeps SQL code separate from business logic.
- How a layered architecture makes it possible to replace one storage mechanism without affecting the rest of the application.

## Reflection

Today's work demonstrated the value of designing software with separation of concerns.

Because the application was already organized into CLI, Service, and Storage layers, replacing JSON with PostgreSQL required changes primarily within the repository and service layers. The CLI continued to function exactly as before, showing how good architecture simplifies future enhancements.

This migration also brought the project closer to becoming a production-ready backend application.
