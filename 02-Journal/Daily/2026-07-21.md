# Daily Journal — 21 July 2026

## Sprint

Sprint 03 — PostgreSQL Migration

---

## Today's Goal

Build the database foundation for the Developer Portfolio Manager by replacing the JSON storage system with PostgreSQL.

---

## What I Worked On Today

Today I began migrating the application from file-based storage to a relational database.

I installed PostgreSQL and configured it for local development. After setting up the PostgreSQL driver (`psycopg2-binary`), I created the `developer_portfolio` database and designed the initial `projects` table.

To keep the application architecture clean, I introduced a dedicated `database` package containing the database connection logic and repository layer. I successfully connected the application to PostgreSQL and verified the setup by executing SQL queries.

Although the application's functionality still uses JSON storage, the database foundation is now ready for migrating each CRUD operation incrementally.

---

## Challenges Faced

- Learning how Python communicates with PostgreSQL through `psycopg2`.
- Designing a clean repository structure without affecting the existing service layer.

## Key Learnings

Today I learned:

## Reflection

Today's work reinforced an important software engineering principle:

> Replacing the storage layer should not require changes to the business logic when the application follows a clean architecture.

Because the project was designed with separate CLI, Service, and Storage layers, introducing PostgreSQL became much more manageable than if all the code had been tightly coupled.
